# Self-Heal Auto-Repair Configuration

This project is configured with a self-adapting auto-repair mechanism designed to automatically fix drift, formatting issues, and test snapshots when they fail in CI or based on an automated schedule.

## How it works

The automation is powered by two main GitHub Actions workflows and three underlying scripts.

### 1. Triggers
The self-healing pipeline (`self-heal.yml`) runs on three triggers:
1. **Scheduled:** Proactively checks and heals code based on a dynamically computed schedule.
2. **Reactive:** Automatically triggers when the main `ci` workflow fails on the default branch.
3. **Manual Dispatch:** Can be manually triggered from the GitHub Actions UI.

### 2. The Repair Pipeline (`scripts/self_heal.mjs`)
When triggered, the script runs the following idempotent repair steps:
1. **Rebuild/reinstall:** Runs `npm ci`.
2. **Lint/format auto-fix:** Runs `eslint --fix` and `prettier -w`.
3. **Snapshot updates:** Updates `vitest` snapshots for tests (excluding natively failing tests in standard environments).
4. **Type stubs:** Re-acquires missing types (extensible if `typesync` is added).
5. **Dependency re-resolve:** Refreshes the lockfile via `npm update`.
6. **Static asset regeneration:** (Placeholder for any static generation scripts).

After the repair steps, a strict gatekeeper (`scripts/healthcheck.mjs`) verifies the integrity of the project. If the healthcheck passes **and** there are file modifications, it will open an auto-repair Pull Request.

### 3. Self-Scheduling (`scripts/compute_schedule.mjs`)
To avoid running unnecessarily, the system observes repository activity (PR merge velocity) and adjusts its own schedule.
The `compute-schedule.yml` workflow runs periodically to calculate a new cadence (ranging from "every 4 hours" for high-velocity projects to "weekly" for dormant ones). If a schedule change is computed, it submits a PR to update the cron expressions.

## Reviewer Checklist

When reviewing a PR generated by this automation:
- Check the **Trigger** (Scheduled, Reactive, or Manual).
- Ensure the logs attached to the workflow run show that the healthcheck failed *before* the fix but passed *after*.
- Verify the changes are restricted to safe paths (`src/`, `tests/`, `package.json`, etc.). The bot will not modify configuration files like `.env` or CI definitions.

## Manual Overrides

If you want to freeze the schedule:
1. Edit `.github/self-heal-schedule.yml` manually.
2. The oscillation guard or manual updates will be respected, but you can also simply modify the `schedule` property.
3. Keep in mind that `compute_schedule.mjs` prevents oscillation by skipping updates if the schedule was modified less than 3 days ago.
