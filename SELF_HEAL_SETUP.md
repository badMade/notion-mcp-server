# Self-Heal Pipeline Setup & Documentation

This repository employs an automated, self-healing pipeline designed to fix CI failures and resolve codebase drift.

## How It Works

The pipeline is triggered in three ways:
1. **Scheduled:** Runs proactively based on a cadence derived from repository telemetry (e.g., PR velocity and CI failure rates).
2. **Reactive:** Triggers automatically if the main `ci` workflow fails.
3. **Manual:** Can be triggered on-demand via Workflow Dispatch.

When triggered, the `scripts/self_heal.mjs` script executes 6 idempotent repair steps in sequence. After each step, a healthcheck (`scripts/healthcheck.mjs`) verifies the state. If the healthcheck passes and there is a diff, the pipeline successfully finishes and creates a PR.

### Repair Steps (Idempotent)
1. **Reinstall:** Performs a clean dependency installation (`npm ci`).
2. **Lint/Format:** Auto-fixes linting and formatting issues (`npx eslint . --fix && npx prettier --write .`).
3. **Snapshots:** Updates testing snapshots (`npx vitest run -u`).
4. **Type Stubs:** Acquires missing types (`npx typesync && npm install`).
5. **Dependency Re-resolve:** Refreshes lockfiles (`npm update`).
6. **Static Asset Regen:** (Skipped/No-op for this repository setup).

## Self-Scheduling

The proactive schedule is not hardcoded. The `scripts/compute_schedule.mjs` script runs periodically (via the `Compute Self-Heal Schedule` workflow) to analyze telemetry (PR merge frequency, CI failure rates) and compute the most optimal interval.

If the calculated interval differs from the current one, the script updates `.github/self-heal-schedule.yml` and creates a PR to update the cron expression in the `.github/workflows/self-heal.yml` workflow.

### Manual Overrides
To override the automated schedule manually, you can edit `.github/self-heal-schedule.yml` directly. Note that the pipeline has an oscillation guard that prevents immediate thrashing. To force the automation to adopt a specific schedule long-term, you may also need to adjust the thresholds in `scripts/compute_schedule.mjs`.

## Reviewer Checklist
When reviewing a PR generated by the self-healing pipeline:
- [ ] Verify that the diff makes logical sense and only contains safe fixes (e.g., snapshot updates, formatting).
- [ ] Confirm no business logic was unintentionally altered.
- [ ] Ensure the PR description matches the trigger type (Scheduled, Reactive, Manual).
- [ ] Check the attached CI logs if necessary.
- [ ] Note that the automation will automatically close stale PRs during subsequent runs.
