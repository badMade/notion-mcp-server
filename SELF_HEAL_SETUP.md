# Self-Heal Auto-Repair Configuration

This repository includes an automated "Self-Heal" pipeline designed to detect code drift, formatting issues, missing snapshots, and out-of-date package lockfiles.

## How It Works

The pipeline is managed by GitHub Actions and Node.js scripts:

1. **`scripts/healthcheck.mjs`**: The gatekeeper. It runs type checks (`tsc`), builds the project, and runs the tests (`vitest`). It exits 0 if the repository is healthy.
2. **`scripts/self_heal.mjs`**: The repair engine. It runs a series of idempotent steps (install, format, update snapshots, dedupe, build) and checks if the healthcheck passes *and* a diff is generated.
3. **`scripts/compute_schedule.mjs`**: The scheduling engine. It looks at the past 14 days of PR merges and CI failures to determine how often the proactive self-heal pipeline should run.
4. **`.github/workflows/self-heal.yml`**: The main workflow. It triggers manually, reactively on CI failure, or proactively on a schedule. It runs the repair script and opens a Pull Request if repairs were made.
5. **`.github/workflows/compute-schedule.yml`**: Runs weekly to update the proactive schedule based on recent telemetry.

## The Repair Steps

When a repair is triggered, it executes these steps in order. After each step, it runs the healthcheck. If the healthcheck passes and there is a diff, it stops early and opens a PR.

1. **Rebuild/reinstall**: `npm ci`
2. **Lint/format**: `npx prettier -w .`
3. **Snapshot updates**: `npx vitest run -u`
4. **Type stubs**: (N/A for this standard TS setup)
5. **Dependency re-resolve**: `npm dedupe`
6. **Asset regeneration**: `npm run build`

## Self-Scheduling

To prevent the pipeline from running too frequently on dormant repositories, or not frequently enough on highly active ones, the pipeline calculates its own optimal cron schedule.

The logic is stored in `scripts/compute_schedule.mjs`. It reads telemetry via the GitHub CLI (`gh`) and categorizes the repo into tiers (High velocity, Active, Standard, Low-churn, Dormant).

The computed schedule and rationale are saved in `.github/self-heal-schedule.yml`.

### Manual Overrides

If you want to freeze the schedule and stop the bot from changing it:

1. Edit `.github/self-heal-schedule.yml`.
2. Remove the `# AUTO-UPDATED` comment from the end of the `schedule` line.
3. Commit and push. The schedule bot relies on that marker to safely perform sed replacements. If missing, the automatic updates to the workflow file will halt, or you can disable the `compute-schedule.yml` workflow entirely.

## Reviewer Checklist

When reviewing a PR generated by this automation:

- [ ] Check the `Files changed` tab. Ensure no sensitive files, secrets, or `.env` files were accidentally committed. (The workflow has safeguards, but human review is always the final gate).
- [ ] Ensure that only formatting, snapshots, lockfile changes, or explicit build artifacts are modified. The script **should never modify source code logic**.
- [ ] Review the rationale if this is a `[Self-Heal Schedule]` PR to ensure the cadence makes sense for your team's workflow.
