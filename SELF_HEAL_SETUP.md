# Self-Healing Pipeline Setup

This project utilizes an automated self-healing CI pipeline configured via GitHub Actions.

## Triggers

1.  **Scheduled:** Runs on an adaptive schedule computed from repository telemetry.
2.  **Reactive:** Runs automatically when the main `ci` workflow fails on the default branch.
3.  **Manual:** Can be dispatched manually via the GitHub Actions UI.

## Repair Pipeline (Order of Operations)

1.  **Reinstall Dependencies:** Clean install of tooling (`npm ci`).
2.  **Lint / Format:** Auto-fixes formatting using Prettier (`npx prettier -w ...`).
3.  **Snapshot Updates:** Regenerates test snapshots (`npx vitest run -u --passWithNoTests`).
4.  **Type Stubs:** (N/A for this Node.js project configuration).
5.  **Dependency Resolution:** Refreshes lockfile.
6.  **Static Assets:** Regenerates built assets (`npm run build`).

## Self-Scheduling Mechanism

A separate workflow (`compute-schedule.yml`) periodically reads the commit history telemetry. If it determines a new optimal cron schedule based on the activity rate, it automatically modifies `.github/workflows/self-heal.yml` and updates `.github/self-heal-schedule.yml`, submitting the change as a PR.

## Overriding the Schedule

If you need to manually set a schedule:
1. Edit `.github/self-heal-schedule.yml` with your desired cron and rationale.
2. The compute schedule script will detect manual modifications and can adapt around them, or you can disable the `compute-schedule.yml` workflow entirely if a strict static schedule is preferred.

## Reviewer Checklist

When reviewing a PR generated by the self-healing automation:
- [ ] Verify that the diff only contains expected changes (formatting, snapshots, lockfiles).
- [ ] Ensure that no source code logic was fundamentally altered.
- [ ] Check that no new dependencies were unexpectedly added outside of user config.
- [ ] Confirm no secrets or `.env` files were accidentally committed.
- [ ] Verify the PR title properly reflects the trigger (Scheduled, Reactive, Manual, or Schedule Update).
