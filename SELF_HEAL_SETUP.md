# Self-Healing CI Pipeline Setup

This document describes the automated self-healing CI pipeline generated for this repository. It runs periodically, reactively, and manually to auto-repair formatting, type, test snapshot, and dependency drift.

## How it Works

The self-healing pipeline triggers via three methods:
1. **Scheduled:** Runs periodically based on an automatically computed schedule.
2. **Reactive:** Triggers whenever the `ci` workflow fails.
3. **Manual:** Can be manually triggered via the Actions tab in GitHub.

When triggered, the pipeline runs the following idempotent repair steps:
- **Step 1:** Rebuild/reinstall (`npm ci`)
- **Step 2:** Lint/format auto-fix (`npx prettier -w ...`)
- **Step 3:** Snapshot updates (`npx vitest run -u`)
- **Step 4:** Type stubs acquisition (`npx typesync` + `npm install`)
- **Step 5:** Dependency lockfile re-resolve (`npm update`)
- **Step 6:** Static asset regeneration (No-op currently)

After each step, a healthcheck runs. If the healthcheck passes and there is a git diff, the script immediately exits and a Pull Request is automatically generated. The PR title varies based on the trigger source.

## Self-Scheduling

The schedule is not hardcoded. Instead, it is computed dynamically by `.github/workflows/compute-schedule.yml`, which runs `scripts/compute_schedule.mjs`.

- The script counts commits over the past 7 days to determine repository velocity.
- It then computes an optimal cron schedule.
- The schedule is written to `.github/self-heal-schedule.yml` using safe YAML updates.
- If the computed schedule differs from the current configuration, a PR is opened to update the workflow.

### Manual Overrides

If you wish to override the schedule manually:
1. Edit `.github/self-heal-schedule.yml`.
2. Edit the schedule block in `.github/workflows/self-heal.yml` under the `# AUTO-UPDATED` comment.
3. The automated process respects existing values if they are manually updated or if the repository goes dormant for an extended period, it will downshift automatically.

## Reviewer Checklist

When reviewing a PR generated by the self-healing automation, check the following:
- [ ] Ensure that code logic was NOT modified (only types, formatting, or lockfiles should change).
- [ ] Verify that no secrets or API keys have been committed.
- [ ] Read the PR description to understand whether the trigger was Proactive, Reactive, or Manual.
- [ ] Confirm the tests and build pipelines continue to pass on the PR branch.
